# Patterns — Universal vs Project-Specific

This document catalogs which patterns belong in the baseline (universal) and which should be generated per-project by bootstrap.

---

## Universal Patterns

These ship in the baseline unchanged. Every project gets them. They are project-agnostic.

### File Size Limits (`docs/rules/coding-standards.md`)

200 soft / 300 hard for source files. 500 for tests. These numbers are the same across every TypeScript project. The reasoning doesn't change based on the framework.

### TDD Workflow (`docs/rules/testing.md`)

Red-green-refactor. Co-located tests. No `it.skip`. Test behavior not implementation. These rules are framework-agnostic — they apply whether you're using Vitest, Jest, or node:test.

### Audit Commands (`.claude/commands/deep-audit.md`, `audit-fix.md`)

The audit format (Critical / High / Medium / Low, with File / Issue / Risk / Fix / Verification / Status fields) is universal. What gets flagged depends on the project, but the structure for recording and fixing findings is the same everywhere.

### Refactor Commands (`.claude/commands/refactor.md`, `refactor-fix.md`)

Same structure as audit commands. What's worth refactoring depends on the project; the workflow for doing it safely doesn't.

### Fix Issue Skill (`.claude/skills/fix-issue/SKILL.md`)

Reproduce → Investigate → Root Cause → Fix → Verify → Commit. This sequence is universal. The tools you use to reproduce (mock context, test DB, etc.) are project-specific — but the skill doesn't name them.

### Implement Feature Skill (`.claude/skills/implement-feature/SKILL.md`)

Understand → Plan → Types → Red → Green → Verify → Checklist → Commit. The TDD loop is universal. The checklist items ("no hardcoded colors," "router delegates to service") are project-specific and handled in `AGENTS.md`.

### Review Skill (`.claude/skills/review/SKILL.md`)

Typecheck, tests, line count, any types, naming, boundaries, dead code. These seven checks are meaningful for any TypeScript project.

### File Size Hook (`.claude/settings.json`)

The hard-fail at 300 / warn at 200 hook is universal. Every project gets it regardless of what other tooling is configured.

## Project-Specific Patterns (generated by bootstrap)

These are not in the baseline — they are generated for each project.

### AGENTS.md

Every project's `AGENTS.md` is different. It describes:
- The specific project architecture and domain
- The specific packages and their boundaries
- The specific commands (`pnpm dev`, `pnpm test`, etc.)
- The specific conventions for that codebase

Bootstrap generates this from reading the project. It is never templated.

`AGENTS.md` is the canonical project-instruction file across all coding agents (Claude Code, Codex, Cursor, Copilot, Gemini CLI, Windsurf, Aider, Zed, Warp, RooCode) per the [agents.md](https://agents.md) open standard. Bootstrap only generates a separate `CLAUDE.md` when the project has Claude-specific overlays that genuinely don't belong in the universal file — most projects skip it entirely and let Claude Code fall back to `AGENTS.md`.

### Formatter/Linter PostToolUse hooks (`.claude/settings.json`)

The file size hook is universal. Everything else (oxfmt hook, Biome hook, oxlint pre-commit) is added by bootstrap after choosing the tooling. Different projects may run different formatters.

### Package Boundary Enforcer (`.claude/agents/package-boundary-enforcer.md`)

The enforcer's structure is universal but its content is entirely project-specific — the allowed dependency graph is different for every monorepo. Bootstrap populates it. Single-package projects delete it.

### Architecture Rules (`docs/rules/[topic].md`)

Project-specific rules like:
- `database.md` — patterns for the specific ORM/DB combination
- `error-handling.md` — how this project handles errors (neverthrow vs try/catch vs Result)
- `styling.md` — token system for this project's design system
- `[framework]-patterns.md` — conventions for Next.js, tRPC, Drizzle, Three.js, etc.

Bootstrap generates these if they're needed. The baseline doesn't include them because they'd be wrong for most projects.

### Per-package `AGENTS.md` (monorepo only)

The agents.md standard supports nested instruction files: the closest `AGENTS.md` in the directory tree wins. Bootstrap generates per-package overlays when a package has non-trivial invariants that don't belong in the root file. Examples: `apps/server/AGENTS.md` capturing "physics state never via Colyseus schema sync"; `packages/db/AGENTS.md` capturing ORM-specific patterns. Keep them short — root rules still apply.

## The Line Between Universal and Project-Specific

Ask: "Would this rule be wrong for a project that doesn't use [specific technology]?"

If yes, it's project-specific. If no, it's universal.

Examples:
- "Files should be under 300 lines" — Universal. No project benefits from longer files.
- "Use Drizzle relational queries with `with:`" — Project-specific. Only relevant if using Drizzle.
- "Write failing tests first" — Universal. Applies regardless of framework.
- "Use `@getpumped/config` instead of `process.env`" — Project-specific. Refers to a specific package.
- "No `any` types" — Universal. TypeScript type safety matters in every project.
- "Import from `packages/core`, not `packages/ui`" — Project-specific. Refers to a specific monorepo structure.

## How the Baseline Evolves

When a pattern that was project-specific turns out to be universal — if 5 different projects independently arrive at the same convention — it can be promoted to the baseline.

Process:
1. Notice the pattern appearing in multiple decision records
2. Extract it into a universal form (remove all project-specific references)
3. Add it to the appropriate universal file
4. Update docs/CHANGELOG.md
5. Future bootstraps get the new universal pattern automatically

This is how the baseline gets smarter over time without becoming opinionated about specific tools.
