- **Structure**: pnpm monorepo (Turborepo) — `packages/core` (domain types + operations), `packages/geometry` (OCCT WASM + Manifold), `packages/state` (Yjs CRDT + Zustand), `packages/ui` (React + Three.js), `packages/api` (tRPC endpoints), `apps/web` (Vite + React)

## Tooling Decisions

### Formatter: Biome

**Why**: Zero-config, fast (sub-10ms), handles TS/JS/JSON/CSS. As a greenfield project, there was no existing formatter to migrate from. Biome's unified format+lint approach eliminates the need to configure two separate tools.

**Considered**:
- oxfmt — very fast for format-only, but project also needed linting
- Prettier — too slow for PostToolUse hooks; more config required

**Agentic fit**: Excellent — fast enough for PostToolUse, zero config to break, unified with linting.

### Linter: Biome

**Why**: Same toolchain as formatter — zero additional config. Runs as part of the same `biome check` command. Since the project chose Biome for formatting, using it for linting avoids managing a separate linting config.

**Considered**:
- oxlint — would have worked alongside Biome format, but adds a second tool to maintain
- ESLint — no compelling need for its plugin ecosystem given the project's stack

**Agentic fit**: Excellent — `biome check` catches both format and lint issues in a single fast pass.

### Test Framework: Vitest + Playwright

**Why**:
- Vitest for unit/integration tests — fast, native ESM, excellent TypeScript support, works seamlessly with Vite's configuration. TDD-first from day 1.
- Playwright for E2E tests — visual snapshot testing for Three.js viewport, browser automation for UI flows.

**Considered**:
- Jest — slower cold start, more config needed for ESM. No compelling reason to choose it over Vitest for a greenfield project.

**Agentic fit**: Excellent — Vitest's watch mode is fast enough for TDD cycles. Clear error messages from both tools.

### Git Hooks: PostToolUse only (no git hook runner)

**Why**: Sole developer using Claude Code. PostToolUse hooks run on every edit — catching issues at edit time rather than commit time. Pre-commit typecheck added directly via PostToolUse PreToolUse hook on `git commit`.

**Considered**:
- Lefthook — would work but adds overhead for a sole-developer project
- Husky — same reasoning

**Note**: The pre-commit TypeScript check (`pnpm typecheck && pnpm test`) runs as a PreToolUse hook before every `git commit`. This is equivalent to a git hook runner but doesn't require installing one.

**Agentic fit**: Excellent for sole developer. PostToolUse hooks catch issues immediately.

### Build: Vite + tsc (kept as designed)

Vite for the web app, tsc for packages. Set at project inception. No changes.

### Package Manager: pnpm (chosen at inception)

pnpm 9.x with workspaces + Turborepo. Set at project inception.

## PostToolUse Hooks Configured

Beyond the universal file size hook:

1. **Biome check** — runs on every TS/TSX file edit, catches format + lint issues simultaneously
2. **Missing test file warning** — warns when a `.ts` source file has no corresponding `.test.ts`
3. **FinalizationRegistry check** — hard blocks use of `FinalizationRegistry` in `packages/geometry/` (must use `LifecycleScope` for OCCT cleanup)
4. **Y.Map check** — hard blocks use of `Y.Map` in `packages/state/` (must use `YKeyValue` — 1900x memory difference)
5. **Throw statement warning** — warns when `throw` appears in non-test source files (must use neverthrow `err()`)

## What Worked / What Didn't

### Worked well

- **Biome unified format+lint**: Zero config to break, fast, catches both categories in one pass. Excellent PostToolUse candidate.
- **TDD from day 1 (greenfield)**: Setting up test infrastructure before any feature code exists eliminates retrofitting. TDD is much harder to introduce to an existing codebase.
- **File size hard ceiling from day 1**: Zero large files inherited. Every file that approached 300 lines got split early, before it became load-bearing architecture.
- **PostToolUse-only hooks for sole developer**: No overhead. The hooks catch issues immediately rather than at commit time.
- **neverthrow Result types**: Effective at eliminating unhandled errors. The `throw` warning hook catches violations before they propagate.
- **Co-located tests** (`foo.test.ts` next to `foo.ts`): Better discoverability. You don't need to navigate to a separate `__tests__/` directory.
- **Package boundary enforcer agent**: Caught real violations early. The read-only constraint is important — the agent reports, human decides.
- **Domain spec docs** (`docs/domain/*.md`): Agent-readable 50-200 line files describing Cabinet Vision concepts. Lets agents understand the domain without reading a 3800-page PDF.

### Didn't work

- Nothing significant — greenfield projects allow you to set conventions before technical debt accumulates.

## Notes

- The OCCT geometry package (WASM-based, no JS fallback) has strict resource lifecycle requirements. The `FinalizationRegistry` ban exists because GC timing is unpredictable for WASM objects — `LifecycleScope` gives deterministic cleanup.
- The Yjs CRDT state layer has a specific constraint: `Y.Map` has a well-known memory issue with large key sets. `YKeyValue` (a third-party abstraction over Yjs) provides the same API with 1900x better memory efficiency for the cabinet/millwork object tree.
- The "AI-agent-native architecture" principle means the API surface is designed so that AI agents and human users are equal clients — no GUI-only functionality. This affects how tRPC procedures are designed and what gets documented.
- The geometry package is the most complex — OCCT WASM operations can crash with degenerate inputs (zero-length edges, coplanar faces, etc.). Tests for geometry functions must include crash-inducing inputs, not just valid ones.
