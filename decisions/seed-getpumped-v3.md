# GetPumped v3 (PumpTrack) — Bootstrap Decisions

> Seeded record — extracted by reading the project, not auto-generated by bootstrap.

## Project Profile

- **Name**: PumpTrack — waste management SaaS for septic pumping companies
- **Type**: Full-stack SaaS (web + mobile)
- **Structure**: pnpm monorepo — `apps/web` (Next.js 16), `apps/pumper` (Expo SDK 55), `packages/api` (tRPC), `packages/db` (Drizzle), `packages/services`, `packages/auth`, `packages/config`, `packages/fieldforms`
- **Size**: Large (~130+ source files, multiple apps)
- **UI**: Next.js (web), React Native / Expo (mobile)
- **Framework**: Next.js 16 (App Router), tRPC v11
- **DB**: Drizzle ORM + PostgreSQL
- **Auth**: Clerk
- **Team**: Multi-developer

## Tooling Decisions

### Formatter: oxfmt

**Why**: Sub-millisecond execution makes it ideal as a PostToolUse hook. Runs after every edit with no perceptible delay. Zero config needed for TypeScript/JavaScript projects.

**Considered**:
- Prettier — too slow (~200ms) for a PostToolUse hook across a large monorepo
- Biome — would have worked, but oxfmt was already established and faster for format-only use case

**Agentic fit**: Excellent — sub-ms speed means the formatter hook adds zero friction to the agent's edit loop.

### Linter: oxlint

**Why**: Complements oxfmt (same Oxc toolchain), fast enough for pre-commit hooks, good TypeScript support.

**Considered**:
- ESLint — rejected for pre-commit use due to speed; too slow to run on every commit in a monorepo
- Biome lint — would have worked well, but the project chose oxfmt so oxlint pairs naturally

**Agentic fit**: Excellent — fast enough to run before every commit without blocking flow.

### Test Framework: Vitest + Jest + Playwright

**Why**: The project uses different frameworks for different layers:
- Vitest for web (Next.js) unit tests — fast, native ESM, good TS support
- Jest (ts-jest) for some packages — kept for compatibility in packages that needed CJS
- Playwright for E2E browser tests — industry standard for browser automation

**Considered**:
- Standardizing entirely on Vitest — tempting but migration cost not justified for working tests

**Agentic fit**: Good — Vitest in particular provides fast feedback for TDD. Jest is slower but acceptable.

### Git Hooks: Lefthook

**Why**: Multi-developer project needs pre-commit gates. Lefthook chosen over Husky for parallel hook execution (faster) and simpler YAML config. Pre-commit runs: `oxlint --deny-warnings && pnpm typecheck`.

**Considered**:
- Husky — widely known, more documentation, but slower sequential execution
- simple-git-hooks — too minimal for a multi-app monorepo
- PostToolUse hooks only — insufficient for team project; team members not all using Claude Code

**Agentic fit**: Good — pre-commit typecheck catches the most common agent mistake (committing broken TypeScript).

### Build: tsc + Turbo (kept existing)

No change. Build tooling was already configured and working. Turbo for monorepo task orchestration.

### Package Manager: pnpm (kept existing)

pnpm 10.x with workspaces. No change.

## PostToolUse Hooks Configured

Beyond the universal file size hook, the following project-specific hooks were added:

1. **oxfmt formatter** — runs after every Edit/Write on TS/JS/JSON files
2. **Package boundary check** — blocks imports from `@getpumped/*` in `packages/fieldforms/` (fieldforms must be standalone)
3. **Router error pattern check** — blocks `throw new Error()` in tRPC router files (must use `handleError()`/`notFound()`/`badRequest()`)
4. **Deep relative import warning** — warns when files use 3+ levels of `../../..` (prefer `@/` aliases)

## What Worked / What Didn't

### Worked well

- **oxfmt PostToolUse hook**: Invisible — formats every file silently, zero friction
- **File size hard-fail at 300**: Effective. Soft warnings were ignored before this change. Hard-fail actually forces splits.
- **Separate audit and fix steps**: Prevented scope creep. Audit produces findings; fix implements one at a time with verification.
- **Pre-commit typecheck**: Catches broken TypeScript before it reaches the repo. Most valuable guardrail.
- **`.claude/file-size-exceptions.txt`**: Works better than listing known-bad files in coding-standards.md — the exceptions are documented and explicit, not embedded in a rule.
- **Package boundary enforcer**: Caught several violations early. Read-only agent pattern works well.

### Didn't work

- **Soft file size warnings**: Were ignored 100% of the time. Hard-fail is necessary.
- **"Known Oversized Files" in coding-standards.md**: Agents would add new exceptions to the list without fixing the underlying issue. Moved to `.claude/file-size-exceptions.txt` where exceptions are more intentional.
- **Audit-and-fix-simultaneously**: Caused regressions when multiple findings conflicted. Separate commands solve this.

## Notes

- The `packages/fieldforms` zero-import boundary is the strictest in the codebase — it's a standalone form builder that must work outside the GetPumped ecosystem. The PostToolUse hook enforcing this has caught real violations.
- tRPC routers using `throw new Error()` was a consistent pattern issue — the hook blocking it has been effective.
- Maestro for Expo E2E tests works well for mobile flows but requires physical/emulator setup — not suitable for CI in all environments.
