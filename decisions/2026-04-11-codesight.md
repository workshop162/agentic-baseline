# codesight — Bootstrap Decisions

> First real consumer of agentic-baseline. Generated by the bootstrap agent (Phase 2 of agentic-baseline plan).

## Project Profile

- **Name**: codesight — drop-in MCP server for context-efficient TypeScript codebase understanding
- **Type**: Library / MCP server (no UI)
- **Structure**: npm workspaces — `packages/codesight` (TypeScript MCP server), `packages/codesight-indexer` (C binary)
- **Size**: Small-medium (~15 source modules, early development)
- **Framework**: None — pure TypeScript Node.js
- **MCP exposure**: 3 tools (`query()`, `exec()`, `index()`)
- **Key dependencies**: @modelcontextprotocol/sdk, better-sqlite3, node-llama-cpp, tree-sitter, zod
- **DB**: SQLite (via better-sqlite3) — knowledge graph, read-only at query time
- **Model**: Gemma 4 E4B via node-llama-cpp, Metal GPU on Apple Silicon
- **Team**: Sole developer

## Tooling Decisions

### Formatter: oxfmt

**Why**: Sub-millisecond execution — ideal as a PostToolUse hook that runs after every Edit/Write. Zero config needed for a TypeScript-only project. The project owner already uses it on GetPumped and has explicit preference for the Oxc toolchain.

**How installed**: `npm install --save-dev oxfmt` in the workspace root. Hook runs `npx oxfmt "$FILE"` on every `.ts`/`.js` file edit.

**Considered**:
- Biome — would also work well, but oxfmt-only is simpler when linting is handled separately by oxlint
- Prettier — too slow (~200ms) for PostToolUse on every edit; not worth the config overhead for a TS-only project

**Agentic fit**: Excellent — sub-ms speed means the hook adds zero friction to the agent's edit loop. Files are always formatted after every write.

### Linter: oxlint

**Why**: Complements oxfmt (same Oxc toolchain), fast enough for pre-commit hooks, good TypeScript support. Since the project uses oxfmt, oxlint is the natural pairing — consistent toolchain, no extra config.

**How installed**: `npm install --save-dev oxlint` in the workspace root. Runs as a PreToolUse hook before `git commit`: `npx oxlint packages/codesight/src --deny-warnings`.

**Considered**:
- ESLint — more rules and plugin ecosystem, but this project doesn't need React/Next.js-specific rules. ESLint's speed and config overhead are not justified here.
- Biome lint — would work if Biome were chosen for formatting; not worth adding a second formatter just to get Biome's lint rules.

**Agentic fit**: Excellent — fast enough to not block commits, catches real issues (unused imports, type errors that tsc misses, etc.).

### Test Framework: Vitest (aspirational — not yet set up)

**Why**: New TypeScript project with ESM (`"type": "module"`) and no existing test framework. Vitest is the natural default: fast, native ESM, compatible Jest API, excellent TypeScript support.

**Status**: No tests exist yet. The testing rule (`.claude/rules/testing.md`) applies and should be followed as the project adds tests. The first test added to this project should set up Vitest.

**What's needed to activate**:
```bash
npm install --save-dev vitest
# Add to packages/codesight/package.json scripts:
#   "test": "vitest run",
#   "test:watch": "vitest watch"
```

**Agentic fit**: Excellent — Vitest's watch mode supports TDD cycles. Fast cold start, clear error messages.

### Git Hooks: PostToolUse + PreToolUse only (no hook runner)

**Why**: Sole developer using Claude Code. PostToolUse hooks (oxfmt, file size check) catch issues at edit time. PreToolUse hook before `git commit` runs typecheck + oxlint. No git hook runner (Lefthook, Husky) needed for a sole developer.

**Hooks configured in `.claude/settings.json`**:
1. **PostToolUse — file size**: Hard-fails if a `.ts` file exceeds 300 lines
2. **PostToolUse — oxfmt**: Formats every `.ts`/`.js` file after Edit/Write
3. **PreToolUse — pre-commit**: Runs `tsc --noEmit` + `oxlint --deny-warnings` before every `git commit`

**Considered**:
- Lefthook — would work; rejected because sole developer with PostToolUse hooks doesn't need a separate git hook runner
- Husky — same reasoning

**Agentic fit**: Excellent for sole developer. Issues caught immediately at edit time rather than deferred to commit time.

### Build: tsc (kept as configured)

No change. The project already uses `tsc` for the TypeScript package and a separate Makefile for the C binary. Both are appropriate for their use case.

### Package Manager: npm (kept as configured)

npm workspaces with `package-lock.json`. No change.

## What Was Considered and Rejected

**Biome (unified format + lint)**: Would have been a reasonable choice for a greenfield project. Rejected because the project owner has explicit preference for the Oxc toolchain (oxfmt + oxlint), and the two are well-documented to complement each other.

**ESLint**: Rejected for linting. No React/Next.js-specific rules needed. Speed and config overhead not justified.

**Lefthook / Husky**: Rejected for sole-developer project. PostToolUse hooks cover the agent's edit loop; PreToolUse before `git commit` covers pre-push validation.

**Jest**: No existing tests, so no migration cost consideration. Vitest is strictly better for a new ESM TypeScript project.

## Agentic Fit Assessment

**High suitability for agentic development** with one important consideration:

**Strengths**:
- Small, well-defined modules with clear boundaries — easy for agents to reason about one module at a time
- No UI layer — no component tree complexity, no visual regression concerns
- Strong confidence model built into the domain — agents can trust calibrated confidence levels from the tool
- Architecture rules are specific and enforceable (the `module-boundary-enforcer` agent can verify them mechanically)
- The `exec()` tool gives agents a direct path to run experiments against the internal APIs

**Consideration — accuracy constraints**:
- The model routing is accuracy-sensitive. An agent that blindly modifies the routing prompt without re-running eval could silently degrade query quality. The accuracy rules in `.claude/rules/accuracy.md` exist to prevent this.
- Strategy results need evidence attached — agents that write "quick" strategies returning only boolean/count results will fail the strategy-reviewer audit.

**Recommendation**:
- Run `module-boundary-enforcer` after any session that touches `strategies/`, `model/`, or `config/`
- Run `strategy-reviewer` when adding or modifying a strategy
- Before changing the routing prompt: always re-run the eval suite and update `docs/MODEL_EVALUATION.md`
